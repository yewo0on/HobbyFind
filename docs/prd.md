# PRD — HobbyFind

## 1. 제품 상세 설명 (Detailed Product Description)

### 1.1 서비스 개요

| 항목 | 내용 |
| --- | --- |
| 서비스명 | HobbyFind |
| 서비스 타입 | 웹 서비스 (반응형 웹, PC/모바일 Web) |
| 목적 | 사용자가 운동형, 지능형, 예술형으로 분류된 취미를 손쉽게 탐색하고, 관심 있는 취미를 북마크하여 관리할 수 있도록 돕는 취미 탐색·관리 서비스 |
| 이용 조건 | 비회원은 간단한 탐색이 가능하나, 회원은 북마크 및 마이페이지를 통해 개인화된 관리 경험을 제공 |

### 1.2 핵심 목표

* **탐색 편의성**: 루트 페이지에서 세 카테고리(운동형·지능형·예술형)를 직관적으로 필터링하고, 취미 카드를 그리드 형태로 빠르게 훑어볼 수 있도록 구성
* **카테고리 집중 탐색**: 카테고리별 전용 페이지에서 해당 유형의 취미만 모아 집중 탐색 가능
* **개인 취미 관리**: 회원이 관심 있는 취미를 북마크로 저장하고, 마이페이지에서 한 번에 확인
* **카테고리별 통계 시각화**: 카테고리별 북마크 분포를 간단한 통계 차트로 가시화
* **교육/실습 적합성**: 취미 목록을 고정하고, 데이터 스키마와 UI가 단순·명료하게 유지되도록 설계하여 강의 수강생들의 프로젝트 실습에 적합하도록 함

### 1.3 취미 카테고리 및 목록 (고정 값)

* **운동형**: 조깅/러닝, 요가, 수영, 자전거, 클라이밍, 댄스
* **지능형**: 독서, 퍼즐, 체스, 프로그래밍, 외국어 학습, 사진 촬영
* **예술형**: 그림 그리기, 악기 연주, 요리, 서예, 도자기 만들기, 정원 가꾸기

※ 위 취미 목록은 **정적 데이터**로 고정되며, **추가/수정/삭제 없음** (교육용 프로젝트 제약 사항).

## 2. 참고 서비스 및 근거 (Reference Services with Detailed Rationale)

| 구분 | 서비스명 | 유형 | 특징 | HobbyFind에 참고하는 포인트 |
| --- | --- | --- | --- | --- |
| 1 | Class101 | 온라인 클래스/취미 플랫폼 | 카테고리별로 정리된 취미/클래스 목록, 카드형 리스트, 상세 페이지 구조 | 카테고리 탭/필터 UX, 그리드 카드 레이아웃 구성 방식 |

| 3 | Meetup | 모임/커뮤니티 플랫폼 | 관심사 기반 카테고리, 로그인 후 개인화된 모임 추천 | 비회원/회원 권한 분리, 로그인 이후의 개인화된 리스트 접근 구조 |

| 5 | 간단한 대시보드/통계 페이지 예제들 | UI 패턴 | 도넛/바 차트 기반 단순 통계 시각화 | 마이페이지 내 카테고리별 북마크 비율을 시각적으로 보여주는 차트 구성 |

**근거 요약**:
* **카드 그리드와 카테고리 필터링**은 Class101, Pinterest 등의 UI 패턴을 참고하여, 취미 탐색의 가시성과 접근성을 높인다.
* **북마크 및 마이페이지 구조**는 북마크 매니저/생산성 도구 패턴을 참고하여, 사용자가 자신의 취미 관심사를 한 곳에서 관리하도록 설계한다.
* **비회원/회원 경험 분리**는 Meetup 등 일반적인 커뮤니티/플랫폼 서비스 패턴을 기반으로, 로그인 전후 기능 차이를 명확히 한다.

## 3. 핵심 기능 및 사양 (Core Features and Specifications)

### 3.1 공통 UI/레이아웃

| 항목 | 내용 |
| --- | --- |
| Top Bar | 좌측 로고, 우측 또는 중앙에 카테고리 필터(운동형, 지능형, 예술형) 버튼 또는 탭 배치 |
| 로고 클릭 | 항상 루트 페이지(`/`)로 이동 |
| 카테고리 필터 클릭 | 해당 카테고리 전용 페이지(`/sports`, `/intellectual`, `/art` 등)로 이동 |
| 반응형 | PC 기준 3~4열, 태블릿 2~3열, 모바일 1~2열 카드 그리드 |

**수용 기준 (Acceptance Criteria)**:
* 로고 클릭 시 항상 홈으로 이동한다.
* 카테고리 필터 클릭 시 해당 카테고리 페이지로 URL이 변경되고, 필터 상태가 시각적으로 강조된다.

---

### 3.2 루트 페이지 (`/`)

**구성 요소**

1. **Top Bar**
   * 로고
   * 카테고리 필터: 운동형 / 지능형 / 예술형 버튼(또는 탭)
   * 로그인/회원가입 (비로그인 시) 또는 마이페이지/로그아웃 (로그인 시)

2. **Hero 섹션**
   * 서비스 소개 문구 (예: "나에게 맞는 취미를 쉽게 찾고, 관리해보세요.")
   * 간단한 서브 텍스트 (예: "운동형, 지능형, 예술형으로 분류된 취미를 한눈에 탐색해보세요.")

3. **메인 콘텐츠 – 취미 카드 그리드**
   * 모든 카테고리를 합친 전체 취미 리스트를 기본으로 노출
   * 카드에는 최소 다음 정보 포함
     * 취미명
     * 카테고리명 (또는 색상 태그/배지)
     * 간단한 설명(한 줄)
     * (회원일 경우) 북마크 토글 버튼(아이콘)

| 항목 | 사양 |
| --- | --- |
| 초기 상태 | 모든 취미(운동형+지능형+예술형)를 하나의 그리드로 노출 |
| 필터링 | Top Bar의 카테고리 클릭 시 해당 카테고리 전용 페이지로 이동 (루트 내 단순 필터링이 아닌, 전용 페이지로 라우팅) |
| 북마크 버튼 | 로그인 상태에서만 활성화/노출 (또는 클릭 가능). 비회원은 비활성/툴팁으로 로그인 요구 문구 표시 가능. |
| 정렬 | 단순 고정 순서(정의된 취미 리스트 순서) 사용. 정렬 기능 추가 없음. |

---

### 3.3 카테고리별 페이지 (`/category/[type]` 또는 `/sports`, `/intellectual`, `/art`)

**구성 요소**

1. **페이지 상단 영역**
   * 선택된 카테고리명 (예: "운동형 취미")
   * 카테고리별 간단 소개 문구 (예: "활동적인 라이프스타일을 위한 운동형 취미를 모았습니다.")

2. **카테고리 전용 취미 카드 그리드**
   * 해당 카테고리에 속한 취미만 그리드로 나열
   * 카드 구성은 루트 페이지와 동일 구조 재사용

| 항목 | 사양 |
| --- | --- |
| 데이터 | 고정 취미 리스트 중 해당 카테고리 항목만 필터링하여 표시 |
| URL 구조 | `/sports`, `/intellectual`, `/art` (예시) – 현재 코드 구조/디자인 기준 유지 |
| 빈 상태 | 항상 최소 1개 이상 취미가 존재하므로 빈 상태 UX 별도 정의 불필요 |

### 3.4 회원 관리 (로그인/회원가입)

**권한 정책**

| 사용자 상태 | 허용 기능 |
| --- | --- |
| 비회원 | 루트 페이지 탐색, 카테고리별 페이지 탐색, 취미 카드 보기 |
| 회원 | 위 기능 + 취미 북마크, 마이페이지 접근, 북마크 리스트 및 통계 조회 |

**회원 관리 기능**

| 기능 | 사양 |
| --- | --- |
| 회원가입 | 아이디(이메일 또는 사용자명), 비밀번호 기반 가입. 최소 유효성 검사(필수 입력, 비밀번호 길이 등). |
| 로그인 | 가입된 아이디/비밀번호로 세션 기반 로그인. 로그인 후 상단에 사용자 메뉴(예: 프로필/마이페이지 링크, 로그아웃 버튼) 노출. |
| 로그아웃 | 상단 메뉴에서 로그아웃 클릭 시 세션 삭제 및 비회원 상태로 전환. |
| 비밀번호 관리 | 본 프로젝트 범위에서는 비밀번호 재설정/찾기 기능은 필수 요구 사항 아님. (추가 기능 아님, 단순히 미구현 명시) |

**수용 기준**:
* 잘못된 아이디/비밀번호 입력 시 에러 메시지를 보여준다.
* 로그인 성공 시 루트 페이지 또는 직전 페이지로 리다이렉트한다.
* 로그인 후 마이페이지 메뉴가 노출되고, 비회원 상태에서는 노출되지 않는다.

### 3.5 북마크 기능

| 항목 | 사양 |
| --- | --- |
| 대상 | 로그인한 회원만 북마크 가능 |
| 인터랙션 | 취미 카드의 북마크 아이콘(예: 별/하트) 클릭 시 토글 (북마크 추가/해제) |
| 상태 표시 | 북마크된 취미는 아이콘 컬러/채움으로 구분 |
| 저장 방식 | 사용자 ID와 취미 ID(또는 고정 키) 조합으로 영구 저장 (DB 또는 영속 저장소) |
| 일관성 | 루트 페이지·카테고리 페이지·마이페이지에서 같은 취미는 동일 북마크 상태를 유지 |

**수용 기준**:
* 로그인 상태에서 북마크 아이콘 클릭 시 페이지 새로고침 없이 상태가 즉시 반영된다.
* 같은 계정으로 재로그인 시, 이전에 북마크한 취미가 그대로 유지되어야 한다.

### 3.6 마이페이지

**경로 예시**: `/mypage` (로그인 필요, 미로그인 시 로그인 페이지로 리다이렉트)

**구성 요소**

1. **북마크 리스트 영역**
   * 사용자가 북마크한 모든 취미를 리스트/그리드 형태로 보여준다.
   * 취미 카드 구성은 루트/카테고리 페이지와 동일하거나 축약 버전(썸네일, 제목, 카테고리, 북마크 해제 버튼) 사용.

2. **카테고리별 북마크 통계 차트 영역**
   * 카테고리(운동형/지능형/예술형)별 북마크 개수 또는 비율을 시각화
   * 예: 도넛 차트(퍼센트), 바 차트(개수) 등 단일 차트
   * 차트에 마우스 오버 시 카테고리명과 수치(개수, 비율) 표시

| 항목 | 사양 |
| --- | --- |
| 레이아웃 | 상단에 요약 타이틀 및 간단 설명, 좌측 또는 상단에 북마크 리스트, 우측 또는 하단에 통계 차트 |
| 정렬 | 북마크 리스트는 북마크한 순서 또는 고정 순서, 별도 정렬/검색 기능 없음 |
| 빈 상태 | 북마크가 하나도 없는 경우 "아직 북마크한 취미가 없습니다." 메시지와 루트/카테고리 페이지로 이동하는 버튼 노출 |

**통계 차트 데이터 예시 구조**:

```json
[
  { "category": "운동형", "count": 3 },
  { "category": "지능형", "count": 2 },
  { "category": "예술형", "count": 1 }
]
```

**수용 기준**:
* 북마크를 추가/삭제하면 마이페이지의 리스트와 차트 데이터가 일관적으로 변경된다.
* 북마크가 없을 때는 차트 대신 "데이터 없음" 상태를 명확히 표시한다.

## 4. 사용자 페르소나 및 시나리오 (User Persona and Scenarios)

### 4.1 사용자 페르소나

| ID | 이름(가명) | 나이/직업 | 목적 | 특성 |
| --- | --- | --- | --- | --- |
| P1 | 지훈(대학생) | 22세 / 대학생 | 학교 생활 외에 운동형 취미를 시작해보고 싶음 | IT 서비스에 익숙, 모바일 중심 사용, 빠른 탐색 선호 |
| P2 | 수연(직장인) | 29세 / 마케터 | 퇴근 후 스트레스 해소용 예술형 취미 찾기 | PC와 모바일 병행, 북마크 후 주말에 다시 확인하는 패턴 |
| P3 | 민수(주니어 개발자) | 26세 / 개발자 | 지능형 취미(프로그래밍, 퍼즐 등) 관심, 체계적으로 정리하고 싶음 | 리스트/통계에 대한 선호도 높음, 정리·관리 기능 중시 |

### 4.2 주요 시나리오

#### 시나리오 S1 – 비회원 탐색

1. 사용자는 브라우저로 HobbyFind 루트 페이지에 접속한다.
2. Hero 섹션에서 서비스 소개 문구를 확인한 뒤, 아래 그리드에서 전체 취미를 빠르게 스크롤하며 살펴본다.
3. Top Bar에서 "운동형" 카테고리 필터를 클릭하여 운동형 취미 전용 페이지로 이동한다.
4. 운동형 페이지에서 각 취미 카드를 보며 간단 설명을 읽고, 마음에 드는 취미가 있는지 탐색한다.
5. 북마크 버튼은 보이더라도 클릭 시 로그인 요구 메시지가 뜨거나 비활성 상태인 것을 확인한다.

#### 시나리오 S2 – 회원 가입 및 취미 북마크

1. 사용자는 북마크 기능을 사용하기 위해 로그인/회원가입 페이지로 이동한다.
2. 간단한 아이디와 비밀번호를 입력해 회원가입을 완료한다.
3. 로그인 후 루트 페이지로 이동하면 상단에 마이페이지 링크와 로그아웃 버튼이 보인다.
4. 사용자는 "예술형" 카테고리 페이지로 이동해, 관심 있는 취미(예: "도자기 만들기", "그림 그리기") 카드에서 북마크 아이콘을 클릭한다.
5. 북마크 아이콘 색상이 변하며 저장된 상태가 시각적으로 표시된다.

#### 시나리오 S3 – 마이페이지에서 북마크 관리 및 통계 확인

1. 로그인된 사용자는 상단 메뉴의 "마이페이지"를 클릭한다.
2. 마이페이지에서 자신이 북마크한 취미 리스트를 확인한다.
3. 필요 없는 취미의 북마크 아이콘을 다시 클릭해 북마크를 해제하면, 리스트에서 해당 카드가 제거되거나 상태가 변경된다.
4. 우측/하단의 통계 차트에서 운동형/지능형/예술형 북마크 비율을 확인하고, 자신의 취미 성향을 한눈에 파악한다.
5. 사용자는 "지능형 취미가 너무 적다"는 것을 인지하고, 지능형 카테고리 페이지로 돌아가 추가로 북마크를 시도한다.

## 5. 기술 스택 추천 (Technical Stack Recommendations)

### 5.1 프론트엔드 및 앱 프레임워크

| 항목 | 추천 스택 | 근거 |
| --- | --- | --- |
| 프레임워크 | **Next.js (최신 안정 버전, 예: 14.x)** | 파일 기반 라우팅으로 루트/카테고리/마이페이지/로그인 페이지 구현이 용이, SSR/SSG 활용 가능 |
| 언어 | **TypeScript** | 교육용 프로젝트로서 타입 안정성과 코드 가독성 향상 |
| UI 라이브러리 | **React** (Next.js 내장) | 컴포넌트 기반으로 카드, 레이아웃, 헤더 등의 재사용이 용이 |
| 스타일링 | **Tailwind CSS** | 유틸리티 클래스 기반으로 빠르게 일관된 디자인 구현 가능, 현재 코드 베이스와 호환 가능성이 높음 |
| 차트 | **Recharts / Chart.js / 기타 React 차트 라이브러리 중 1개** | 마이페이지 카테고리별 북마크 통계 시각화에 사용 (간단한 바/도넛 차트) |

### 5.2 백엔드/데이터 및 인증

| 항목 | 추천 스택 | 근거 |
| --- | --- | --- |
| 백엔드 | Next.js API Routes 또는 Route Handlers | 소규모 프로젝트에 충분, 별도 서버 없이 API 구현 가능 |
| 데이터베이스 | PostgreSQL / SQLite / Supabase 등 | 교육용이므로 간단한 스키마(Users, Bookmarks 테이블)로 구성 가능 |
| ORM | Prisma | 타입 안전한 DB 접근, Next.js/TypeScript와 궁합이 좋음 |
| 인증 | Credentials 기반 인증 (예: NextAuth.js) | 아이디/비밀번호 기반 로그인 구현 용이, 세션 관리 내장 |

### 5.3 기타

* **상태 관리**: 페이지 전환과 간단한 북마크 액션에는 **React Query** 또는 Next.js 서버 액션 + 클라이언트 상태 최소화로도 충분.
* **배포**: Vercel 등 Next.js 친화적인 플랫폼을 권장 (교육용 배포에 적합).

## 6. 제한사항 (Limitations)

| 구분 | 내용 |
| --- | --- |
| 취미 데이터 | 취미 목록은 명시된 고정 리스트만 사용하며, 관리자 UI나 동적 추가/수정/삭제 기능은 제공하지 않는다. |
| 기능 범위 | 소셜 기능(댓글, 공유, 팔로우 등), 추천 알고리즘, 고급 검색/필터링 기능은 포함하지 않는다. |
| 회원 관리 | 비밀번호 찾기/재설정, 소셜 로그인 등 고급 인증 기능은 범위 밖으로 둔다. |
| 스케일 | 교육용/실습용 프로젝트를 전제로 하며, 대규모 트래픽/다국어 지원/접근성(A11y) 최적화 등은 최소 수준으로만 고려한다. |
| 데이터 보안 | 기본적인 비밀번호 해싱 등은 필요하지만, 실제 상용 수준의 보안·법적 요건(예: 개인정보 처리 방침, 이메일 검증 등)은 필수 범위로 포함하지 않는다. |
